# 設計ドキュメント

## 背景

高性能な長文脈モデルでも、調査・設計・実装・レビューを1つの会話に詰め込むと、
前提・目的・制約・途中結果が混ざり合い、精度が落ちる。一方で「とにかくサブエージェントに
分割する」のも誤りで、起動のたびにコールドスタートのコスト（文脈の再導出・遅延・トークン）を払う。

このテンプレートは、その中間にある実用解を狙ったもの:
**サブエージェント分業を「コンテキスト汚染対策」として限定的に使い、文脈の共有はチャットではなくファイルで行う。**

## 3原則の詳細

### 1. サブエージェントはコンテキスト汚染対策

- 使う場面: 大量のファイル読解・ログ解析・検索結果・長い定型実装など、メイン文脈を埋める作業
- 使わない場面: 短いタスク、小修正、単発質問（コールドスタートのコスト・遅延・再説明の負担が上回る）
- これは Claude Code 公式のサブエージェント設計思想（メイン会話を検索結果やログで汚さない）と一致する

### 2. ファイルベース文脈共有

チャットのメモリ・セッション文脈はツール間で共有できない。そこで成果物を必ずディスクに固定する:

| ファイル | 役割 |
|---|---|
| `docs/refactor-plan.md` | `/analyze-codebase` の成果物。Phase 分けされた計画 |
| `docs/handoff/HANDOFF-<topic>.md` | `/handoff` の成果物。外部ツール向け自己完結実装指示書 |
| `AGENTS.md` | ツール横断で共有する前提（Codex は作業前にこれを読む） |

ハンドオフ文書の品質基準は1つ:
**「このセッションを一切知らない実装者（Codex / Cursor / 別セッション / 人間）が、この文書だけで作業を完了できること」**

### 3. コスト階層ルーティング

| 担当 | モデル | 割り当て | 理由 |
|---|---|---|---|
| PM / 設計 | メイン会話（上位モデル） | 設計判断・統合・重いアーキテクチャ判断 | 判断の質が成果を支配する工程 |
| implementer | Sonnet 固定 | 単純・定型・大量編集 | 判断が単純なら上位モデルは過剰 |
| implementer-heavy | Opus 固定 | 複数ファイル横断・中複雑度 | 軽い設計判断までを任せる中間層 |
| reviewer | メイン継承 | 検収 | 品質ゲートには高い判断力を残す（コスト優先なら Sonnet に変更可） |
| 外部ツール | Codex / Cursor | usage limit 逼迫時・長時間定型実装 | レート制限の分散 |

ポイントは「上位モデルは実装が苦手だから渡す」のではなく、
**高価な判断力を設計・検収という高付加価値工程に温存する**という費用対効果の設計であること。

## 安全設計

- **reviewer は読み取り専用**: `tools: Read, Grep, Glob, Bash` で Write/Edit を除外。Bash は検査系コマンド（git status / diff / log、テスト実行等）に限定する制約を本文に明記。ただしこれはプロンプトレベルの制約であり完全な権限分離ではない — 厳格運用では Bash を外しテストはメイン会話で実行する
- **自動委任ガード**: グローバル配置のエージェントは全プロジェクトに影響するため、description に「明示的に任されたときだけ使う」を日英併記し、不要な自動委任を抑制
- **空引数・誤走査ガード**: コマンドは空 `$ARGUMENTS` で停止して確認を求める。`/analyze-codebase` はホームディレクトリや `C:\` 直下など巨大ツリーからの実行を拒否する
- **検収必須の範囲を限定**: reviewer 検収が必須なのは (1) 外部ツールから戻った差分、(2) `/implement-phase` の成果、の2つのみ。それ以外の小差分は任意（「短いタスクにサブエージェントを使わない」原則を優先）

## 検証プロセス

このテンプレート自体を多段で検証した。

### 1. 敵対的レビュー（読み取り専用エージェント3体）

観点別に並列実行:

| 検証担当 | 観点 |
|---|---|
| agents-spec | frontmatter 構文・model ID・tools 構文・自動委任リスク |
| commands | 空引数時の挙動・失敗モード・自己完結性 |
| consistency | グローバルルールとの矛盾・ルーティング表と実体の不一致 |

結果: **findings 18件（critical 3 / warning 6 / info 9）**。

### 2. 採用した修正（6件）

| 問題 | 修正 |
|---|---|
| `/implement-phase` が空引数のまま進む（critical） | Phase 番号の確認と実在チェックを追加 |
| `/handoff` が空引数で `HANDOFF-.md` を生成（critical） | トピック名確認＋Windows-safe slug 正規化＋ディレクトリ作成を追加 |
| reviewer がグローバルルール（MCP ツール優先）と矛盾（critical） | 「このエージェントに MCP ツールはない。native ツールを使う」と明記 |
| description の委任ガードが日本語のみ（warning） | 英語ガードを併記 |
| 巨大ツリーからの全走査リスク（warning） | プロジェクトルート確認を追加 |
| 「検収必須」と「過剰委任禁止」の緊張（warning） | 検収必須の範囲を2ケースに限定 |

### 3. スモークテスト（3体全合格）

| エージェント | 結果 |
|---|---|
| implementer | 起動成功・指定モデルで解決・読み取りのみで完了（変更ゼロ） |
| implementer-heavy | 同上 |
| reviewer | 指示した git 検査コマンド3つのみ実行・変更ゼロ・制約遵守 |

full model ID（`claude-sonnet-4-6` / `claude-opus-4-8`）が有効であることも確認済み。

## 既知の限界

- 「一括実行 vs サブエージェント分離」の定量比較は未実施（設計は公式のサブエージェント思想との整合に基づく）
- 外部ツール側の挙動（AGENTS.md の解釈・ハンドオフ文書の読み方）は各ツールの仕様に依存
- 自動委任の頻度は実運用での継続観察項目
